## 1. Why the current data model strains at world scale

Today's data is organized into **region files** registered in `data/index.json`:
`canada.json` (1 country), `usa.json` (1 country), `europe.json` (33 countries),
`ancient.json` (a *theme*, 10 countries). Every page calls `Destinations.loadAll()` which
fetches **every** region file on load and flattens them.

This worked at 4 files. It breaks down going global, for four concrete reasons:

1. **The "region" is an inconsistent unit.** It's sometimes a country (Canada), sometimes a
   continent (Europe = 33 countries crammed in one file), sometimes a cross-cutting *theme*
   (Ancient Apocalypse, whose sites are physically in the USA, Mexico, Peru, Turkey…).
   Mixing geography and theme in one registry is a category error — and it means a single
   place can't cleanly roll up to "its country."
2. **Country is not a first-class entity.** It's a free-text string on each location, so it
   drifts: `England` vs `United Kingdom` for places in the *same* country (see Issues §I-1).
   There's no canonical list, no guaranteed flag/continent/ISO code.
3. **No continent dimension to browse by.** At 89 countries, users need
   Continent → Country → City navigation. Today there's no continent on most records
   (canada has none; usa/europe have it at the file's top level only).
4. **Load-everything doesn't scale.** Fetching every country's full JSON on every page load
   is fine at 191 locations; at thousands it's wasteful. HANDOFF/PLAN already flag a deferred
   `data/locations-index.json` manifest — the overhaul is the moment to land it.

## 2. Proposed architecture — country-first, with a lightweight index

Three ideas, all doable in vanilla JS with no bundler:

### (a) Country becomes a first-class registry
New `data/countries.json` — the canonical list (drives filters, flags, the backlog, the map):
```jsonc
{ "countries": [
  { "code":"JP", "name":"Japan", "flag":"🇯🇵", "continent":"Asia",
    "file":"data/places/jp.json", "enabled":true, "status":"backlog" },
  { "code":"GB", "name":"United Kingdom", "flag":"🇬🇧", "continent":"Europe",
    "file":"data/places/gb.json", "enabled":true, "status":"live" }
]}
```
`status: "live" | "backlog" | "stub"` lets the backlog from `COUNTRIES.md` live *in data*, so
the map can even show "coming soon" countries honestly (principle #2) rather than hiding them.

### (b) Separate GEOGRAPHY from COLLECTIONS
- **Geography** = Continent → Country → Place. The default browse axis.
- **Collections** (Ancient Apocalypse, and future themes) = a **`collections` array on the
  location** (`"collections": ["ancient"]`), a cross-cutting *tag*, NOT a region. An Ancient
  site in Mexico is then simply a Mexican place that's *also* tagged `ancient` — it rolls up
  to Mexico for geography AND shows under the Ancient filter. This kills the double-counting
  (today USA shows 56 because 6 Ancient-US sites are mixed into the modern 50) and makes the
  next themed collection free (matches the existing data-driven `accent` mechanism).

### (c) A slim generated index, lazy-load detail
- `tools/build_index.py` generates `data/index.geo.json`: one tiny record per place
  (`id, name, country code, continent, coordinates, flags: {walk,window,kind}`) — just enough
  for the **map, search, City Guesser shortlist, and Window chip row**.
- Full per-place detail (blurb, highlights, culture, sounds, street_view…) is fetched **per
  country, on demand** when you open a place or filter to that country.
- Result: home map loads one small index instead of every country file; detail is pay-as-you-go.

**Reorganize files by country** (`data/places/<iso>.json`) so adding a country = drop one file +
one registry row. `europe.json`'s 33 countries get split; `ancient.json` dissolves into a
`collections` tag on the relevant country files.

### What stays the same (deliberately)
Vanilla HTML/CSS/JS, no framework, no bundler. `tools/*.py` already establish the
"generate data with a script" pattern — `build_index.py` is one more of those. The scene
player, Window resolver, radio, soundscape, passport — all untouched; they consume the same
flat location array `Destinations.loadAll()` returns today (we just change how it's assembled).

## 3. Migration path (incremental, ship-as-you-go — no big bang)

The whole point is to avoid a rewrite that breaks the working app. Order:

1. **Add `data/countries.json`** (registry of all 89, `status` field) — pure addition, nothing
   reads it yet. Seed `status:"backlog"` rows straight from `COUNTRIES.md` §4.
2. **Normalize country labels in place** (fix England→UK etc., add `country_code` +
   `continent` to every record) via a one-off `tools/normalize_countries.py`. Low risk, high
   payoff — fixes Issues §I-1/I-2 immediately, independent of everything else.
3. **Introduce `collections` tag**; migrate `ancient.json` sites into their host-country files
   tagged `["ancient"]`; retire `ancient` as a "region". Verify the Ancient filter still works
   off the tag.
4. **Split `europe.json` → per-country files**; update `data/index.json`→`countries.json`.
   `Destinations.loadAll()` changes only its *source list*, not its output shape.
5. **Generate `data/index.geo.json`** + switch home map/search/guesser to it; flip detail to
   lazy per-country fetch. (This is the only step that touches page JS meaningfully.)
6. **Then** start filling the backlog country-by-country (Tier 1 first: Japan, Australia,
   Russia, China, India, Argentina, Thailand, Malaysia, South Korea).

Steps 1–2 are worth doing **now** regardless of the rest — they're cheap and fix real bugs.

## 4. Issues / bugs / cleanup log (found while mapping the project)

Severity: 🔴 correctness · 🟡 hygiene/tech-debt · 🔵 nice-to-have.

- **I-1 ✅ FIXED — inconsistent country labels in `europe.json`.** `London`+`Edinburgh` were
  `"United Kingdom"` while `Liverpool/Oxford/Manchester/Cambridge/Jurassic Coast` were `"England"`
  (and `Edinburgh` is in Scotland, so `"England"` was outright wrong). `normalize_countries.py`
  folded all constituents into `"United Kingdom"` with the constituent kept as `region`.
- **I-2 ✅ FIXED — no country_code / continent on records.** Every location now carries
  `country_code` (ISO alpha-2) + `continent`, stamped from the registry by `normalize_countries.py`.
- **I-3 🟡 OPEN — `ancient` region conflates theme with geography.** Its 25 sites are physically in
  10 countries but siloed in `ancient.json`. *Not yet* converted to a `collections:["ancient"]`
  tag — deferred (it touches map/passport/guess JS that read the region registry). The registry's
  `live_counts` already attributes ancient sites to their real countries, so country counts are
  honest; only the *file/region* grouping remains theme-shaped. Next structural step.
- **I-4 ✅ FIXED — Faroe Islands counted as sovereign.** Now `country:"Denmark", region:"Faroe
  Islands"`. The pin is kept; the count is honest. Same rule will apply to Gibraltar/Greenland/etc.
- **I-5 🟡 OPEN — `Destinations.loadAll()` fetches every region file on every page load.** Now 8
  files / 345 locations — still fine, but the case for the slim `data/index.geo.json` + lazy detail
  grows with every region. Deferred (proposal §2c).
- **I-6 ✅ MOSTLY FIXED — top-level `country` field drift.** Per-record country is now canonical
  everywhere; the *file-level* top `country` field is now vestigial (the registry is the source of
  truth). Harmless.
- **I-7 🔵 OPEN — Window `onError` fallback still missing** (carried from PLAN/HANDOFF §9 #1). A
  rotted cam/ambient id shows a broken YouTube frame instead of the honest empty state. Unchanged
  by this work.
- **I-8 🔵 Environment note (not a project bug).** In this sandbox `PATH` is stripped, so
  `curl`/`node`/`python3` must be called by absolute path (`/usr/bin/...`); and the
  `claude-in-chrome` extension was offline (curl+grep recon fallback used).
- **I-9 🟡 NEW — new countries are coordinates-only; content depth is thin.** The 154 new
  destinations have verified coords + a valid `wikipedia_slug` (so photos/summary/weather/time all
  resolve live), but **no authored `highlights`, no curated `walk`/`webcam`/`ambient`, and `blurb`
  falls back to Wikipedia.** That's an honest baseline, not a finished one. Follow-ups: curate
  windows/walks for marquee new cities (Tokyo, Sydney, Dubai…), author highlights.
- **I-10 🟡 NEW — culture phrases/dishes need a native-speaker review.** `build_culture.py`'s ISO
  codes, currencies, and capitals are factual; the **greeting phrases and signature dishes are
  best-effort** (native scripts where confident, romanized where risky). Have a native/fluent
  reviewer pass over the 50 new entries before treating them as authoritative.
- **I-11 🔵 NEW — stale culture keys.** `'England'` and `'Faroe Islands'` remain as keys in
  `culture.js`'s four tables but no location uses them anymore (folded into UK/Denmark). Harmless
  dead weight; drop on the next `culture.js` edit.

## 5. How this was gathered (reproducible)

Browser extension was offline, so (per HANDOFF §7) I pulled raw HTML with `curl` and parsed the
inline JS arrays with `node`:
```bash
# fetch (absolute path because sandbox PATH is stripped)
for p in "" livecam window videarth; do
  /usr/bin/curl -sL -A "Mozilla/5.0" -o "vv_${p:-home}.html" "https://virtualvacation.us/$p"
done
# /videarth: var countriesAndCities = [['City','Country','Region'], …]  → 985 pts, 496 cities, 84 countries
# /window:   var locations = [['City','<ytid>?start=ssssss'], …]        → 117 city-level entries
# /livecam:  var location_lst = ['<ytid>', …]                           → 30 live-cam ids (no names in source)
```
Extract + diff scripts (`extract.js`, `diff.js`, `continents.js`) and their JSON outputs
(`extracted.json`, `country_diff.json`) are in the session scratchpad. They diff
virtualvacation's `countriesAndCities` countries against the `country` field across our four
`data/*.json` files, with the normalization rules in `COUNTRIES.md` §5.

## 6. Build log / toolchain (re-runnable, in order)

The whole expansion is reproducible from four new Python tools (same "generate data with a
script" pattern as the existing `build_europe.py` etc.). Run in this order:

```bash
python3 tools/build_countries.py     # → data/countries.json  (registry; counts read from index.json)
python3 tools/normalize_countries.py # → rewrites canada/usa/europe/ancient.json (UK fold, +code/continent)
python3 tools/build_world.py         # → asia/africa/oceania/latinamerica.json + appends to europe.json
                                     #   (verifies every coord/slug via Wikipedia API; cache: tools/.wiki_cache.json)
python3 tools/build_culture.py       # → patches js/culture.js (4 tables × 50 countries)
python3 tools/build_countries.py     # re-run to flip the 50 backlog → live now that data exists
```

Notes / gotchas hit while building (so the next run doesn't re-learn them):
- **`build_world.py` uses the MediaWiki action API**, not the REST summary endpoint. Two real traps,
  both handled: the `coordinates` prop **paginates** (must follow the `continue` token, or pages past
  the first chunk look coord-less) and some cities store coords as **secondary** (`primary:false`,
  needs `coprimary=all`). It also honours `Retry-After` on 429s with escalating backoff.
- **The original `build_europe.py`/`build_usa.py` generators are one-off and now STALE** (ambient
  entries + the normalization were applied to the live JSON directly). Don't re-run them — they'd
  clobber `country_code`/`continent`/ambient and re-introduce `England`. Treat the live JSON as source.
- `normalize_countries.py` and `build_world.py` (europe append) are **idempotent**. Backups of the
  pre-change data were taken in the session scratchpad (`data_backup/`).
- Verified post-build: 345 locations / 89 countries, 0 duplicate ids, all coordinates in range,
  `Destinations.loadAll` loader simulation passes, `Culture.get/currencyCode/facts` functional, all
  `js/*.js` pass `node --check`.
